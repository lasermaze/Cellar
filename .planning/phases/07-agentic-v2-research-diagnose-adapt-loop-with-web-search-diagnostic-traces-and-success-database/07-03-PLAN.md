  <action>
1. Add `trace_launch` and `check_file_access` to `execute()` switch and `toolDefinitions`.

2. **trace_launch ToolDefinition:**
   - name: "trace_launch"
   - description: "Run a short diagnostic Wine launch with debug channels enabled. The game is killed after timeout_seconds. Returns structured DLL load analysis (which DLLs loaded, from where, native vs builtin) and any errors detected. Use this BEFORE configuring — trace first, then fix. Does NOT count toward the launch limit."
   - Optional params: `debug_channels` (array of strings, default ["+loaddll"]), `timeout_seconds` (number, default 5)

3. **trace_launch implementation** (`private func traceLaunch(input: JSONValue) -> String`):
   - Extract debug_channels (default ["+loaddll"]) and timeout_seconds (default 5)
   - Build environment: copy accumulatedEnv, set WINEDEBUG by merging channels with any existing value
   - Create Process with wineProcess.wineBinary, arguments [executablePath], environment including WINEPREFIX
   - Set `process.currentDirectoryURL` to binary's parent directory (matches WineProcess CWD fix)
   - Use Pipe for stderr with StderrCapture (reuse the same @unchecked Sendable + NSLock pattern from WineProcess)
   - Create kill timer: `DispatchWorkItem { process.terminate(); try? self.wineProcess.killWineserver() }` scheduled via `DispatchQueue.global().asyncAfter(deadline: .now() + Double(timeoutSeconds))`
   - Start process, waitUntilExit, cancel kill timer
   - Parse stderr for +loaddll lines using regex: `trace:loaddll.*Loaded L"([^"]+)" at [0-9A-Fa-f]+: (native|builtin)`
   - Extract DLL name (last path component), full path, type (native/builtin)
   - Deduplicate by DLL name (keep last occurrence — Wine may load/unload/reload)
   - Also extract any Wine error lines (err: prefix)
   - Return JSON: `loaded_dlls` array of {name, path, type}, `errors` array of strings, `timeout_applied` bool, `raw_line_count` int
   - IMPORTANT: Do NOT return raw stderr — only structured parsed output (prevents context flooding per Pitfall 3)

4. **check_file_access ToolDefinition:**
   - name: "check_file_access"
   - description: "Check if the game can find files it needs by verifying file existence relative to the game executable's directory. Use to diagnose 'file not found' errors caused by wrong working directory."
   - Required param: `paths` (array of strings — relative paths to check)

5. **check_file_access implementation** (`private func checkFileAccess(input: JSONValue) -> String`):
   - Extract paths array from input
   - For each relative path: check existence from game exe's parent directory
   - Also check from bottleURL/drive_c as fallback context
   - Return JSON: array of {path, exists_from_game_dir: bool, absolute_path: string}

<task type="auto">
  <name>Task 2: verify_dll_override + enhanced inspect_game</name>
  <files>Sources/cellar/Core/AgentTools.swift</files>
  <action>
1. Add `verify_dll_override` to `execute()` switch and `toolDefinitions`.

2. **verify_dll_override ToolDefinition:**
   - name: "verify_dll_override"
   - description: "Verify that a DLL override is actually working by comparing the configured override (env/registry) with what Wine actually loaded (via a short trace). Explains discrepancies like 'native DLL exists in game_dir but Wine loaded builtin from syswow64'."
   - Required param: `dll_name` (string — e.g. "ddraw")

3. **verify_dll_override implementation** (`private func verifyDllOverride(input: JSONValue) -> String`):
   - Extract dll_name
   - Collect current override config: check accumulatedEnv["WINEDLLOVERRIDES"] for the dll
   - Check if native DLL file exists in game directory, system32, syswow64
   - Run a short trace launch internally (call `traceLaunch` with input `{debug_channels: ["+loaddll"], timeout_seconds: 3}`) — parse the result JSON string back to extract loaded_dlls
   - Find the DLL in the loaded_dlls list
   - Compare: configured override vs actual load path vs actual load type
   - Build explanation string describing any discrepancy:
     - "Override set to n,b but Wine loaded builtin — native DLL may not be in the right location"
     - "Native DLL found in game_dir but Wine loaded from syswow64 — for system DLLs in wow64 bottles, place in syswow64"
     - "Override working correctly: configured n,b, loaded native from game_dir"
   - Return JSON: `configured_override`, `native_dll_locations` (array), `actual_load` (path + type), `explanation`, `working: bool`

4. **Enhanced inspect_game:**
   Update the existing `inspectGame(input:)` method to add three new data points:

   a. **PE imports via objdump -p**: After the existing `/usr/bin/file` check, spawn `/usr/bin/objdump -p <executablePath>`. Parse output for lines matching `DLL Name: <name>` prefix. Collect into `pe_imports: [String]` array. If objdump fails or no imports found, return empty array (defensive — per Pitfall 2). Also try fallback: scan for lines containing ".dll" if "DLL Name:" format not found.

   b. **Bottle type detection**: Check if `bottleURL/drive_c/windows/syswow64` directory exists. If yes, `bottle_type: "wow64"`, else `bottle_type: "standard"`.

   c. **Data files listing**: Read the game directory (exe's parent) and list files with extensions: .dat, .ini, .cfg, .txt, .xml, .json (up to 50 files). Add as `data_files: [String]` array. This helps the agent find config files like mode.dat.

   d. **Known shim detection**: Check PE imports against a list of known shim DLLs: ["ddraw.dll", "d3d8.dll", "d3d9.dll", "d3d11.dll", "dinput.dll", "dinput8.dll", "dsound.dll"]. Flag any matches as `notable_imports` with a note about what they mean (e.g., "ddraw.dll — DirectDraw game, consider cnc-ddraw").

   Add all new fields to the result dictionary.
