# Phase 1.1: Reactive Dependencies - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning
**Source:** Investigation of Phase 1 pipeline failure (dotnet48 deadlock during `cellar add`)

<domain>
## Phase Boundary

Fix the dependency installation pipeline in AddCommand and extend LaunchCommand's retry loop. Currently, `cellar add` blindly pre-installs winetricks deps from `recipe.setupDeps` before running the game installer. This caused a 30+ minute dotnet48 install that deadlocked Wine's NGen compilation — for a 2001 DirectDraw game that doesn't need .NET at all.

The system already has WineErrorParser (detects missing DLLs, maps them to winetricks verbs) and a retry loop in LaunchCommand. The reactive diagnosis system works but is bypassed by proactive pre-installation.

This phase changes the architecture: try first, diagnose on failure, install only what's actually needed.

</domain>

<decisions>
## Implementation Decisions

### Dependency Strategy
- AddCommand runs installer WITHOUT pre-installing setup_deps
- If installer fails with diagnosable error (WineErrorParser): install just that dep, retry
- If installer fails with no diagnosis but recipe.setupDeps exists: offer recipe deps as fallback (user prompt)
- `--force-proactive-deps` flag restores old pre-install behavior for power users
- Recipe `setup_deps` field semantic changes from "must pre-install" to "known fixes if needed"

### WinetricksRunner Service
- Extract winetricks execution from AddCommand inline code into reusable service
- Track `lastOutputTime` in readability handlers — kill if no output for 5 minutes (stale-output detection)
- Reuse NSLock wrapper pattern from WineProcess.StderrCapture for thread safety
- Always pass `-q` (unattended mode) to winetricks
- Kill wineserver on timeout to clean up Wine processes
- Return structured result: verb, success, timedOut, exitCode, elapsed

### LaunchCommand Extension
- When WineErrorParser suggests installWinetricks fix: install dep via WinetricksRunner, retry SAME config
- Don't advance to next env variant until dep fix has been tried
- Track which deps were installed to avoid re-installing on subsequent attempts
- Cap total attempts at 5 (across both dep installs and variant cycling)

### WineResult Extension
- Add `timedOut: Bool` field to WineResult
- Existing WineProcess.run() returns timedOut: false for backward compatibility

### Bug Fixes (while here)
- AddCommand.swift:36-37: Fix redundant nil check (compiler warning)
- WineProcess.swift: Set readabilityHandler = nil after drain (consistency with GuidedInstaller)

### Claude's Discretion
- Exact timeout durations (5 min stale suggested, implementer can adjust)
- Whether to add tests for WinetricksRunner (depends on testability)
- Internal error handling flow details

</decisions>

<specifics>
## Specific Ideas

### Root Cause (from investigation)
The Cossacks recipe specified `"setup_deps": ["dotnet48"]` based on research into GOG installer requirements that was never validated through end-to-end testing. The dotnet48 winetricks install spawns NGen compilation workers that deadlock under Wine Crossover on ARM, creating a 60-second retry loop that runs indefinitely.

### Key Files
- `Sources/cellar/Commands/AddCommand.swift` — lines 49-99 are the inline winetricks loop to replace
- `Sources/cellar/Commands/LaunchCommand.swift` — lines 81-146 are the retry loop to extend
- `Sources/cellar/Core/WineErrorParser.swift` — already has dllToWinetricksFix() mapping
- `Sources/cellar/Core/WineProcess.swift` — has StderrCapture pattern to reuse, needs static killWineserver
- `Sources/cellar/Models/WineResult.swift` — add timedOut field
- `recipes/cossacks-european-wars.json` — keep setup_deps as hint, no changes needed

### Existing Patterns to Reuse
- `WineProcess.StderrCapture` (NSLock + @unchecked Sendable) — for thread-safe lastOutputTime tracking
- `GuidedInstaller.runStreamingProcess()` — readability handler + drain + nil handler pattern
- `WineErrorParser.dllToWinetricksFix()` — already maps DLL names to winetricks verbs
- `WineProcess.killWineserver()` — already exists for SIGINT handling

</specifics>
