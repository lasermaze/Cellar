# Phase 1.1 Plan 02: Reactive Dependency Pipeline Summary

**Installer runs first without pre-installing deps; WinetricksRunner installs diagnosed deps on failure, and LaunchCommand retries same Wine config before cycling variants — eliminating the 30-minute dotnet48 deadlock.**

## Accomplishments

- Removed 60+ lines of inline Process code from AddCommand's winetricks loop; replaced with WinetricksRunner calls
- AddCommand now runs installer first; only installs winetricks deps reactively if installer fails with a diagnosis
- User prompted for recipe setup_deps as fallback when installer fails with no diagnosis
- --force-proactive-deps flag restores old pre-install behavior for power users
- LaunchCommand while-loop retry retries same envConfig after dep install, only advancing configIndex when no dep fix was applied
- installedDeps Set prevents re-installing the same verb across attempts
- Total launch attempts capped at 5 (maxTotalAttempts) across dep installs and variant cycling

- `Sources/cellar/Commands/LaunchCommand.swift` - While-loop retry with dep install before variant cycling; installedDeps Set; maxTotalAttempts=5

## Decisions Made

- setup_deps semantic change: from "must pre-install" to "known fixes if installer fails" — the core motivation of this phase, eliminates the 30+ minute dotnet48 pre-install that caused the deadlock
- AddCommand fallback prompts user before installing recipe setup_deps (no-diagnosis case) — respects user agency when we have no specific diagnosis
- LaunchCommand does NOT advance configIndex when a dep was successfully installed — retries same config with dep in place before trying a different Wine variant
- maxTotalAttempts=5 covers both dep installs and variant cycling combined, replacing the old min(count, 3) cap
