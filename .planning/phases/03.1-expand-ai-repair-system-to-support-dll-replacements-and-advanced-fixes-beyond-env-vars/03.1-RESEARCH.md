## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Fix Action Vocabulary
- Expand WineFix to cover **everything Wine can do**: env vars, DLL overrides, DLL file replacements, registry edits, winetricks verbs, winecfg changes, prefix settings, exe flags
- AI can suggest **multi-step compound fixes** — e.g., "install cnc-ddraw AND set registry key AND add env var" as a single variant
- Winetricks allowlist expanded but maintained — unknown verbs get a warning to user: "AI suggested verb X which isn't in our known list — proceed?" rather than silent rejection
- New fix actions should map to expanded WineFix enum cases

#### Graduated Escalation
- **Level 1**: Env vars and WINEDLLOVERRIDES only (current Phase 3 behavior)
- **Level 2**: Add DLL overrides + winetricks verb installs
- **Level 3**: DLL file replacements + registry edits + everything else
- Each level sends a **different prompt** with more action types available — progressive prompt expansion
- Keep **maxTotalAttempts=10** — graduated escalation changes what each attempt can do, not how many attempts
- Show escalation progression to user: "Level 1: trying env var tweaks... Level 2: trying DLL overrides... Level 3: installing cnc-ddraw + registry edits..."

#### DLL Acquisition Strategy
- **Download from GitHub releases** — AI knows the project, Cellar fetches the release artifact
- Place DLLs in **game directory** next to the executable (how cnc-ddraw works — game loads local DLL before Wine's system version)
- **Cache downloaded DLLs** in ~/.cellar/dlls/ for reuse across games
- **Known DLLs only** for auto-download — curated registry of open-source DLL replacement projects (cnc-ddraw, dgVoodoo, dxwrapper, etc.) with GitHub repo URLs and release patterns
- AI can suggest unknown DLLs but Cellar won't auto-download them — tells user where to get it manually
- **Trust GitHub HTTPS** for integrity — no separate hash verification needed
- **Download failures: skip and continue** repair loop — treat as failed variant, don't block the loop
- **Fully automatic** — no user confirmation for downloads (consistent with auto-apply behavior)

#### cnc-ddraw Special Support
- **Both first-class AND discoverable** — bundled recipes for known DirectDraw games include cnc-ddraw; AI can also discover and recommend it for unknown DirectDraw games
- cnc-ddraw is the primary motivation for this phase: Cossacks: European Wars (2001 DirectDraw) fails with "DirectDraw Init Failed (80004001)"

#### Safety Boundaries
- **Action type allowlist** — Cellar only executes known action types. Unknown action types logged and skipped.
- **Sandbox to WINEPREFIX** — all file operations constrained to the game's WINEPREFIX directory. DLL downloads go to cache (~/.cellar/dlls/), placements go to game dir inside prefix.
- **Accept persistence** — registry edits and DLL placements persist even if variant fails. No bottle snapshotting.
- Both guardrails apply simultaneously

#### AI Prompt Evolution
- **Structured action schema** — AI returns typed actions: {type: "place_dll", source: "cnc-ddraw", target: "game_dir"}, {type: "registry", key: "...", value: "..."}, etc.
- **Progressive prompt expansion** — each escalation level sends a different prompt with more action types available
- **Game-specific context** in prompt — include game era, DirectX version, installer type, Wine translation stack context
- **Reasoning drives the fix** — AI reasons first, then derives actions FROM the reasoning. Show reasoning to user before each attempt.

## Summary

Phase 03.1 expands the AI repair system from its current Level 1 capabilities (env vars + WINEDLLOVERRIDES only) to a three-tier graduated escalation system that can also download and place DLL replacements, run winetricks verbs, and write registry entries. The primary motivation is cnc-ddraw support for Cossacks: European Wars, which needs a custom ddraw.dll placed in the game directory.

The implementation breaks into three concrete subsystems: (1) an expanded `WineFix` enum with new cases for `placeDLL`, `setRegistry`, and compound actions; (2) a `DLLDownloader` service that knows how to fetch from GitHub releases latest API and cache locally; (3) a progressive prompt system in `AIService.generateVariants()` that receives an escalation level parameter and returns differently-scoped action schemas. The three-level escalation is driven by attempt count buckets within the existing `maxTotalAttempts=10` budget.

The technical surface is well-understood from direct code inspection. The existing patterns (NSLock + @unchecked Sendable, synchronous URLSession with semaphore, `WinetricksRunner` for verbs, `WineProcess.applyRegistryFile()`) all compose cleanly with the new capabilities. No new Swift Package Manager dependencies are needed — zip extraction uses `/usr/bin/ditto` or `/usr/bin/unzip` via `Process`, both available on all macOS systems.

**Primary recommendation:** Implement in three logical plans: (1) expand WineFix enum + new action executor infrastructure; (2) DLL download/cache/place system with known DLL registry; (3) progressive prompt system + escalation loop integration + Cossacks recipe update.

### Pattern 1: Expanded WineFix Enum
**What:** Add new cases to the existing `WineFix` enum in WineErrorParser.swift
**When to use:** Every new action type from AI or WineErrorParser

```swift
enum WineFix {
    // Existing (Level 1)
    case installWinetricks(String)              // verb name
    case setEnvVar(String, String)              // key, value
    case setDLLOverride(String, String)         // dll, mode

    // New (Level 2)
    case placeDLL(String, DLLPlacementTarget)   // dllName, target

    // New (Level 3)
    case setRegistry(String, String, String)    // key, valueName, data (raw .reg format)
    case installWinetricksMulti([String])       // multiple verbs in sequence

    // Compound (any level)
    case compound([WineFix])                    // ordered list of sub-actions
}

enum DLLPlacementTarget {
    case gameDir    // next to EXE (how cnc-ddraw works)
    case system32   // Wine's virtual System32 (less common, within WINEPREFIX)
}
```

### Pattern 6: WineActionExecutor
**What:** New struct to dispatch `WineFix` actions, separating execution from LaunchCommand
**Why:** LaunchCommand is already long. Extracting action execution keeps the loop readable.

```swift
struct WineActionExecutor {
    let wineProcess: WineProcess
    let wineURL: URL
    let bottleURL: URL
    let gameDir: URL   // game executable's directory inside bottle

    // Execute a single WineFix. Returns true if action succeeded.
    // Failures: print warning, return false (loop continues)

```

### Anti-Patterns to Avoid
- **Adding ZIPFoundation or other SPM deps**: Use `/usr/bin/ditto` instead — zero dependencies
- **Blocking on download failure**: Download failures skip and continue (per decisions) — never throw
- **Persistent re-download**: Check `~/.cellar/dlls/cnc-ddraw/ddraw.dll` exists before hitting network
- **Shell-expanding DLL paths**: Always use `Process` with argument arrays, never shell strings (established codebase pattern)
- **Modifying `envConfigs[configIndex]` for DLL actions**: DLL placements and registry edits are side effects, not env changes. Track them separately in `appliedActions` for repair report.
- **Confusing action schema with variant schema**: Level 1 variants can still use the old `environment` field for backward compat with bundled `retry_variants` in recipes. New AI variants use `actions` array.

| Problem | Don't Build | Use Instead | Why |

## Common Pitfalls

### Pitfall 1: cnc-ddraw needs BOTH dll placement AND WINEDLLOVERRIDES
**What goes wrong:** Placing ddraw.dll in game dir is not enough. Wine must be told to prefer native DLL over builtin.
**Why it happens:** Without `WINEDLLOVERRIDES=ddraw=n,b`, Wine may use its own builtin ddraw even with a local file present.
**How to avoid:** The `place_dll` action for cnc-ddraw should always be accompanied by `set_dll_override` action. Encode this in the known DLL registry or in the AI prompt's example.
**Warning signs:** cnc-ddraw placed but game still shows DirectDraw error.

### Pitfall 2: GitHub API rate limiting
**What goes wrong:** Unauthenticated GitHub API calls are limited to 60/hour per IP. In rapid testing, this can be hit.
**Why it happens:** No auth token in requests.
**How to avoid:** Cache is the primary defense — once downloaded, never re-fetch. The `cachedDLL` check before any network call means repeated launches cost 0 API calls.
**Warning signs:** HTTP 403 with `X-RateLimit-Remaining: 0`.

### Pitfall 3: DLL caching invalidation
**What goes wrong:** Cache has old version of cnc-ddraw; new version has fixes.
**Why it happens:** No TTL or version check in cache.
**How to avoid:** Per decisions, no hash verification is needed. Cache invalidation is deferred (v2). For v1, cached version is always used if present — acceptable trade-off.

### Pitfall 4: Escalation without retry for level 2/3 actions
**What goes wrong:** Level 3 action (DLL placement) is applied but the attempt is immediately counted as failed because the fix requires a fresh launch.
**Why it happens:** DLL placements are side effects that persist. The current attempt's launch result doesn't reflect the fix.
**How to avoid:** After applying a DLL placement or registry edit, set `depInstalled = true` (the existing reuse pattern in LaunchCommand) to trigger a retry of the same attempt. The fix is persistent, so the next launch will benefit from it.

### Pitfall 5: actionSchema backward compatibility
**What goes wrong:** Bundled `retry_variants` in recipe JSON still use `environment` field (not `actions`). Parsing breaks.
**Why it happens:** AI now returns `actions` array, but RecipeEngine still loads recipes with `environment` fields.
**How to avoid:** Keep `RetryVariant.environment` as the primary field for recipe-level variants. Add `RetryVariant.actions` as an optional field. LaunchCommand applies `actions` when present, falls back to `environment` otherwise.

### Pitfall 6: Game directory path inside WINEPREFIX
**What goes wrong:** cnc-ddraw needs to be placed next to the game EXE, but `executablePath` is an absolute host path, not relative to the bottle.
**Why it happens:** `entry.executablePath` is stored as a full macOS path (set during `cellar add`).
**How to avoid:** Derive `gameDir` as the parent directory of `executablePath`. This is the correct location — for GOG Cossacks it would be `~/.cellar/bottles/cossacks-european-wars/drive_c/GOG Games/Cossacks - European Wars/`.

### Pitfall 7: dgVoodoo2 does not work on Wine
**What goes wrong:** AI might suggest dgVoodoo2 for DirectX games. Including it in the known DLL registry would cause failed downloads and placements.
**Why it happens:** dgVoodoo2 relies on D3D11 which Wine/Proton doesn't support the same way.
**How to avoid:** Do NOT include dgVoodoo2 in the known DLL registry for v1. cnc-ddraw and dxwrapper are the safe choices.

### parseWineFix for new action types
```swift
// Extend existing parseWineFix() in AIService
// New action schema uses "type" field instead of "fix_type"
case "place_dll":
    let dllName = dict["dll"] as? String ?? ""
    let target = dict["target"] as? String ?? "game_dir"
    guard !dllName.isEmpty else { return nil }
    return .placeDLL(dllName, target == "game_dir" ? .gameDir : .system32)
```

## DirectX Version Detection (Claude's Discretion item)

For detecting DirectX version from error logs (not binary scanning), the most reliable signal is the error text itself:
- `"DirectDraw Init Failed"` or `"ddraw"` in stderr → DirectDraw (DX1-7, ddraw.dll)
- `"d3d8"` in import errors → DirectX 8
- `"d3d9"` in import errors → DirectX 9

For file scanning approach: scan game directory for `ddraw.dll`, `d3d8.dll`, `d3d9.dll` as a hint. If game ships its own `ddraw.dll` (broken), cnc-ddraw placement will be overridden by game's version — flag this to user.

**Recommendation:** Include error log pattern matching in WineErrorParser for DirectDraw errors (`"DirectDraw Init Failed"` maps to `.placeDLL("cnc-ddraw", .gameDir)` suggestion). File scanning is a secondary enrichment for the AI prompt context.

## Open Questions

1. **Should `RetryVariant` in Recipe.swift grow an `actions` field?**
   - What we know: Recipe.swift currently only has `environment: [String: String]`
   - What's unclear: Adding `actions` to `RetryVariant` enables bundled recipes to specify DLL placements; but requires Recipe decoder update
   - Recommendation: Yes — add `actions: [[String: AnyCodable]]?` or `actions: [Data]?` using a typed wrapper. Necessary for the Cossacks recipe update.

2. **How many escalation levels map to attempt numbers?**
   - What we know: maxTotalAttempts=10, bundled variants exhausted at configIndex >= originalCount
   - What's unclear: Whether 3/3/4 split (attempts 1-3/4-6/7-10) is the right cadence
   - Recommendation: Use configIndex-based tracking rather than totalAttempts for level: Level 1 for first AI call (after bundled variants exhausted), Level 2 if Level 1 variants all fail, Level 3 if Level 2 also fails. More natural than attempt-count buckets.

3. **How to encode registry edits in `WineFix.setRegistry`?**
   - What we know: WineProcess.applyRegistryFile() takes a .reg file URL
   - What's unclear: Whether to store raw .reg format string or parsed key/value/data
   - Recommendation: Store as structured (key, valueName, data) tuple. Format into .reg string at execution time. More composable for AI to generate than raw .reg format.
