# Phase 03.1: Expand AI Repair System - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Expand the AI repair system beyond env vars and WINEDLLOVERRIDES to support DLL replacements (cnc-ddraw), registry edits, winetricks verbs, and other advanced Wine fixes. The AI model's reasoning — not hardcoded restrictions — should drive what fixes are attempted. Graduated escalation ensures simpler fixes are tried first, with progressively more powerful actions unlocked as simpler approaches fail.

Requirements: None (INSERTED phase — extends RECIPE-04 capabilities)

</domain>

<decisions>
## Implementation Decisions

### Fix Action Vocabulary
- Expand WineFix to cover **everything Wine can do**: env vars, DLL overrides, DLL file replacements, registry edits, winetricks verbs, winecfg changes, prefix settings, exe flags
- AI can suggest **multi-step compound fixes** — e.g., "install cnc-ddraw AND set registry key AND add env var" as a single variant
- Winetricks allowlist expanded but maintained — unknown verbs get a warning to user: "AI suggested verb X which isn't in our known list — proceed?" rather than silent rejection
- New fix actions should map to expanded WineFix enum cases

### Graduated Escalation
- **Level 1**: Env vars and WINEDLLOVERRIDES only (current Phase 3 behavior)
- **Level 2**: Add DLL overrides + winetricks verb installs
- **Level 3**: DLL file replacements + registry edits + everything else
- Each level sends a **different prompt** with more action types available — progressive prompt expansion
- Keep **maxTotalAttempts=10** — graduated escalation changes what each attempt can do, not how many attempts
- Show escalation progression to user: "Level 1: trying env var tweaks... Level 2: trying DLL overrides... Level 3: installing cnc-ddraw + registry edits..."

### DLL Acquisition Strategy
- **Download from GitHub releases** — AI knows the project, Cellar fetches the release artifact
- Place DLLs in **game directory** next to the executable (how cnc-ddraw works — game loads local DLL before Wine's system version)
- **Cache downloaded DLLs** in ~/.cellar/dlls/ for reuse across games
- **Known DLLs only** for auto-download — curated registry of open-source DLL replacement projects (cnc-ddraw, dgVoodoo, dxwrapper, etc.) with GitHub repo URLs and release patterns
- AI can suggest unknown DLLs but Cellar won't auto-download them — tells user where to get it manually
- **Trust GitHub HTTPS** for integrity — no separate hash verification needed
- **Download failures: skip and continue** repair loop — treat as failed variant, don't block the loop
- **Fully automatic** — no user confirmation for downloads (consistent with auto-apply behavior)

### cnc-ddraw Special Support
- **Both first-class AND discoverable** — bundled recipes for known DirectDraw games include cnc-ddraw; AI can also discover and recommend it for unknown DirectDraw games
- cnc-ddraw is the primary motivation for this phase: Cossacks: European Wars (2001 DirectDraw) fails with "DirectDraw Init Failed (80004001)" through Wine + Rosetta + MoltenVK translation stack

### Safety Boundaries
- **Action type allowlist** — Cellar only executes known action types (set env var, place DLL, run winetricks, set registry key, etc.). AI can't invent arbitrary shell commands. Unknown action types logged and skipped.
- **Sandbox to WINEPREFIX** — all file operations constrained to the game's WINEPREFIX directory. Nothing outside the bottle is modified. DLL downloads go to cache (~/.cellar/dlls/), placements go to game dir inside prefix.
- **Accept persistence** — registry edits and DLL placements persist even if variant fails. No bottle snapshotting. User can reset bottle (Phase 4) if needed. Matches how Wine users actually work.
- Both guardrails (allowlist + sandbox) apply simultaneously — belt and suspenders

### AI Prompt Evolution
- **Structured action schema** — AI returns typed actions: {type: "place_dll", source: "cnc-ddraw", target: "game_dir"}, {type: "registry", key: "...", value: "..."}, etc.
- **Progressive prompt expansion** — each escalation level sends a different prompt with more action types available. Level 1 prompt lists only env vars. Level 2 adds DLL overrides + winetricks. Level 3 adds everything.
- **Game-specific context** in prompt — include game era (2001), DirectX version (DirectDraw/DX8), installer type (GOG), plus general Wine translation stack context. Maximizes AI's chance of relevant fixes.
- **Reasoning drives the fix** — AI reasons first about the problem, then derives actions FROM the reasoning. Reasoning is the primary output; actions are the structured consequence. Show reasoning to user as the AI "thinking" before each attempt.

### Claude's Discretion
- Exact JSON schema for the expanded action types
- How to parse GitHub release pages to find the right artifact
- Known DLL registry format and initial contents
- How escalation levels map to attempt numbers within maxTotalAttempts=10
- Prompt engineering for each escalation level
- How to detect which DirectX version a game uses from error logs/file scan
- Whether to add new AIService methods or restructure existing generateVariants()

</decisions>

<specifics>
## Specific Ideas

- The trigger for this phase: Cossacks: European Wars gets "DirectDraw Init Failed (80004001)" because Wine's ddraw implementation can't initialize through Wine + Rosetta + MoltenVK (4 translation layers). cnc-ddraw replaces the broken ddraw.dll with one that works on modern systems. The Phase 3 AI could reason about this but couldn't act — it was constrained to env vars by design.
- "A reasoning of AI model is exactly what should help us running games, not env restrictions" — the AI's knowledge of Wine compatibility IS the value. Artificial constraints on what it can suggest limit its usefulness.
- REQUIREMENTS.md already has RECIPE-06 (cnc-ddraw integration) in v2 — this phase partially delivers it as part of the expanded fix system.
- The repair report (Phase 3) already captures what was tried. With expanded fix types, reports become more valuable — showing DLL replacements and registry edits attempted.

</specifics>

- NSLock + @unchecked Sendable wrapper for thread-safe state (WineProcess, WinetricksRunner)
- Structured JSON AI responses with explicit schema in system prompt
- AI silent on .unavailable — no API key is not an error
- Attempt history with 500-char cap per entry for prompt token management
- Retry 3 times on API failure, fall back gracefully

### Integration Points
- `LaunchCommand.swift` retry loop: graduated escalation changes which prompt/tools are available at each stage
- `AIService`: generateVariants() prompt needs progressive expansion per escalation level
- New DLL download + cache system (~/.cellar/dlls/)
- Known DLL registry (new data structure — maps DLL names to GitHub repos + release patterns)
- Bundled Cossacks recipe: add cnc-ddraw as first-class fix
