**Graduated AI escalation (env vars -> winetricks -> DLL replacements) wired end-to-end: LaunchCommand now dispatches all WineFix types through WineActionExecutor and calls AI at 3 escalating power levels, with cnc-ddraw as the first Cossacks retry variant**

## Accomplishments

- generateVariants() now accepts escalationLevel 1/2/3 with distinct system prompts per level — Level 1 prohibits registry/DLL/winetricks, Level 2 adds winetricks and set_dll_override, Level 3 adds place_dll and set_registry
- LaunchCommand replaces single aiVariantsGenerated boolean with a 3-level escalation counter; exhausting each level immediately tries the next without stopping
- All fix dispatch (error-diagnosed, AI-diagnosed, variant-level actions) goes through WineActionExecutor.execute() — ~40-line inline switch removed
- Cossacks recipe now has cnc-ddraw as first retry variant with place_dll action; recipe actions parsed through AIService.parseWineFix(from:) at load time
- Repair report includes Applied Actions section tracking DLL placements and registry edits

## Decisions Made

- AIVariant (runtime, has parsed [WineFix]) is distinct from RetryVariant (Codable, for recipes) — AI variants at runtime need parsed fix objects, not the JSON-representable type
- WineActionExecutor.execute() tuple signature updated rather than adding an overload — the actions field is now always present, consistent across all callers
- Escalation loop uses `continue` when a level produces no variants, immediately trying the next level without stopping
- `parseWineFix(from:)` made internal (not private) so LaunchCommand can call it directly for parsing recipe-level actions at load time
- Winning config serializes WineFix actions back to [[String:String]] for RetryVariant.actions persistence — round-trips cleanly
