**Sources/cellar/Commands/AddCommand.swift** — Replace the stub:

```swift
struct AddCommand: ParsableCommand {
    static let configuration = CommandConfiguration(
        commandName: "add",
        abstract: "Add a game by running its installer"
    )

    @Argument(help: "Path to the game installer (e.g., setup.exe)")
    var installerPath: String
}
```

`run()` flow:
1. Verify `installerPath` exists using FileManager. If not, print error and exit.
2. Check dependencies: `DependencyChecker().checkAll()`. If `!allRequired`, print "Run `cellar` first to install dependencies." and exit.
3. Derive game ID from installer's parent directory name (per user decision: "Game name auto-derived from directory name"). Slugify: lowercase, replace spaces with hyphens, strip non-alphanumeric except hyphens. Example: "Cossacks European Wars" -> "cossacks-european-wars".
4. Check if game already exists in CellarStore. If so, print "Game already added. Use `cellar launch \(gameId)` to play." and exit.
5. Print: "Adding game: \(gameName)"
6. Create bottle: `BottleManager(wineBinary: status.wine!).createBottle(gameId: gameId)`
7. Print: "Running installer inside Wine bottle..."
8. Run the GOG installer inside the bottle using WineProcess:
   - `wineProcess.run(binary: installerPath, arguments: ["/VERYSILENT", "/SP-", "/SUPPRESSMSGBOXES"])`
   - Pass installer arguments as array (NEVER as shell string — RESEARCH.md Pitfall 2: backslash escaping)
   - Stream output in real-time
9. After installer exits, save game entry to CellarStore:
   ```swift
   let entry = GameEntry(
       id: gameId,
       name: gameName,        // human-readable from directory name
       installPath: "...",    // will be determined by recipe executable
       recipeId: gameId,      // assume recipe ID matches game ID
       addedAt: Date()
   )
   CellarStore.addGame(entry)
   ```
10. Print: "Game added successfully. Run `cellar launch \(gameId)` to play."

Note on `installerPath`: The user points `cellar add` at the installer (setup.exe), not pre-installed files (per CONTEXT.md decision). The installer runs inside the bottle and installs the game there.

**Sources/cellar/Commands/LaunchCommand.swift** — Replace the stub:

```swift
struct LaunchCommand: ParsableCommand {
    static let configuration = CommandConfiguration(
        commandName: "launch",
        abstract: "Launch a game with Wine"
    )

    @Argument(help: "Game name (as shown in cellar status)")
    var game: String
}
```

`run()` flow — this is the most complex command, implementing the full launch pipeline:

1. **Check dependencies**: `DependencyChecker().checkAll()`. If missing, print error and exit.

2. **Find game**: `CellarStore.findGame(id: game)`. If not found, print "Game not found. Run `cellar add /path/to/installer` first." and exit.

3. **Check bottle exists**: `BottleManager(wineBinary: status.wine!).bottleExists(gameId: game)`. If not, print error suggesting `cellar add` again.

4. **Load recipe**: `RecipeEngine.findBundledRecipe(for: game)`. If found, proceed. If not found, print warning "No recipe found for \(game). Launching with default Wine settings." (Graceful — relevant for Phase 2 AI recipes).

5. **Apply recipe** (if found):
   - Create `WineProcess(wineBinary: status.wine!, winePrefix: CellarPaths.bottleDir(for: game))`
   - `let recipeEnv = try RecipeEngine().apply(recipe: recipe, wineProcess: wineProcess)`
   - This prints the transparent registry/env changes

6. **Prepare log file**:
   - Create log directory: `FileManager.default.createDirectory(at: CellarPaths.logDir(for: game), withIntermediateDirectories: true)` (Pitfall 7)
   - Open log file at `CellarPaths.logFile(for: game, timestamp: Date())`
   - Print: "Log: \(logFilePath)"

7. **Set up SIGINT handler** (per CONTEXT.md: Ctrl+C kills Wine process but still asks validation):
   ```swift
   signal(SIGINT, SIG_IGN)
   let sigintSource = DispatchSource.makeSignalSource(signal: SIGINT, queue: .main)
   sigintSource.setEventHandler {
       gameProcess.terminate()
       // Don't exit — let the normal post-exit flow handle validation
   }
   sigintSource.resume()
   ```

8. **Launch game**:
   - Determine executable path: recipe's `executable` field gives the relative path inside the bottle's `drive_c/` directory. The actual path depends on where the GOG installer placed the game (typically `C:\GOG Games\Cossacks\` or similar). Construct the Wine-side path.
   - Print: "Launching \(game)..."
   - Record `startTime = Date()`
   - Call `wineProcess.run(binary: executablePath, arguments: recipe?.launchArgs ?? [], environment: recipeEnv ?? [:], logFile: logFileURL)`
   - This streams Wine output to terminal AND log file simultaneously (per CONTEXT.md: "Wine stdout/stderr streams to terminal in real-time")

9. **Post-exit flow**:
   - Calculate `elapsed = Date().timeIntervalSince(startTime)`
   - Cancel SIGINT source: `sigintSource.cancel()`
   - Restore default SIGINT: `signal(SIGINT, SIG_DFL)`
   - Run `ValidationPrompt.run(gameId: game, elapsed: elapsed, wineProcess: wineProcess)`
   - If result is non-nil, update game entry: `entry.lastLaunched = Date(); entry.lastResult = result; CellarStore.updateGame(entry)`
   - Print: "Launch result recorded."

**Important implementation notes:**
- The game executable path inside the Wine bottle needs to be a Windows-style path (e.g., `C:\\GOG Games\\Cossacks\\dmln.exe`) or a Unix path under the bottle's `drive_c/`. Wine accepts Unix paths to files inside the prefix. Use the Unix path: `{bottlePath}/drive_c/GOG Games/Cossacks - European Wars/{recipe.executable}`.
- The exact install directory depends on the GOG installer behavior. For Phase 1, hardcode the expected GOG install path for Cossacks. If it differs in practice, this is an easy fix.
- Process arguments MUST be passed as a Swift array, never interpolated into a shell string (Pitfall 2).
