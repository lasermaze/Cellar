# v1.1 End-to-End Test Manual

Test the full v1.1 milestone: loop resilience, engine detection, dialog detection, and smarter research. One test session covers almost everything.

## Before you start

Make sure you have your Anthropic API key set:

```bash
echo $ANTHROPIC_API_KEY
```

Build the project:

```bash
cd ~/Documents/Cellar
swift build
```

Delete the Cossacks bottle so the agent starts fresh — this is important because engine pre-configuration only happens on the first launch:

```bash
rm -rf ~/.cellar/bottles/setup-cossacks-european-wars-21013
```

Re-create the empty bottle so Wine has somewhere to work:

```bash
swift run cellar add setup_cossacks_european_wars_2.1.0.13.exe
```

## Test 1: Full agent session with budget tracking

Set a low budget so you can see the budget system in action. $0.50 is enough for a real session but low enough that you'll see the 50% console alert:

```bash
CELLAR_BUDGET=0.50 swift run cellar launch setup-cossacks-european-wars-21013
```

Now watch the output. You're looking for these things in order:

### Engine detection (Phase 9)

Early in the session, the agent will call `inspect_game`. In the output, you should see it identify the engine. Look for something like:

- Engine detected as **GSC/DMCR** (or similar)
- Graphics API detected as **DirectDraw** (from ddraw.dll or mdraw.dll imports)
- Confidence level (high, medium, or low)

If the agent just says "inspecting game files" and moves on without mentioning engine type, something is wrong.

### Pre-configuration before first launch (Phase 9)

Before the agent calls `launch_game` or `trace_launch` for the first time, it should set up the game. Look for:

- The agent placing **cnc-ddraw** DLL files into the game directory
- The agent writing a **ddraw.ini** file with renderer settings
- The agent possibly writing registry entries

The key thing: this should happen **before** the first launch attempt, not after a failed launch. The whole point is that the agent knows what this engine needs and configures it proactively.

### Dialog detection (Phase 10)

When the agent launches the game, look at the result it gets back:

- The launch result should include a **dialogs** array (even if empty — that's fine if no dialogs appeared)
- If a dialog does appear (like a renderer selection or error box), the `dialogs` array should contain the message text
- The agent may also call **list_windows** to check what's on screen — look for it reporting window sizes and whether any look like small dialog boxes

### Research quality (Phase 11)

If the agent needs to research fixes (because something didn't work on first try), watch for:

- When it calls `fetch_page` on a WineHQ or PCGamingWiki URL, the result should include **extracted_fixes** with specific items (env vars, DLL names, registry paths) — not just a wall of text
- When it calls `query_successdb`, it might use **similar_games** to find fixes from games with the same engine — look for similarity scores in the results

### Budget tracking (Phase 8)

As the session runs:

- Around 50% of your $0.50 budget, you should see a **console message** about budget usage (just a print, not something the agent sees)
- Around 80%, the agent itself gets a **warning message** injected into its context
- At 100%, the agent gets **one final call** to wrap up, then the session stops

At the very end, regardless of how the session ends, you should see:

```
Session cost: $X.XX (N input + M output tokens, K iterations)
```

Verify the numbers aren't all zeros.

### Search query quality (Phase 9)

If the agent searches the web, look at the search queries it constructs. They should include the engine name and graphics API, like:

> "GSC DMCR DirectDraw Wine macOS black screen"

Not just:

> "Cossacks European Wars Wine macOS"

## Test 2: Error recovery (Phase 8)

This is harder to trigger deliberately, but keep an eye out during the session:

- If the API returns an error, you should see the agent **retry** (up to 3 times with increasing delays) rather than immediately crashing
- If a response gets cut off mid-tool-call (max_tokens truncation), the agent should **retry with a larger token limit** instead of sending broken JSON
- If the agent gets an empty response, it should **send a continuation prompt** ("Please continue...") rather than silently stopping

You probably won't see all of these in a normal session. They're safety nets. If the session completes normally, that's good — it means nothing went wrong. If something does go wrong and the agent recovers gracefully, that's great — it means the safety nets work.

## Test 3: Screen Recording permission (Phase 10)

This tests the graceful degradation of window detection:

1. Open **System Settings → Privacy & Security → Screen Recording**
2. Make sure your terminal app is **not** in the allowed list
3. Run a launch session
4. When the agent calls `list_windows`, it should still work — returning window bounds and process names, but **no window titles**
5. The result should include `screen_recording_permission: false`
6. The agent should mention the permission limitation **once** and then carry on using Wine trace data only

If you grant Screen Recording permission and run again, `list_windows` should now include window titles.

## Test 4: Cross-game similarity (Phase 11)

This tests whether the agent can find fixes from similar games. Since Cossacks is already in the success database, try launching a different DirectDraw game:

```bash
swift run cellar launch setup-deus-ex-goty-1112fmrevision-1630-51757
```

Watch for the agent calling `query_successdb` with `similar_games` parameters (engine type and graphics API). It should find the Cossacks entry as a similar game and potentially apply similar fixes — since both are old DirectDraw games.

## What success looks like

A fully working v1.1 session looks like this:

1. Agent starts, loads budget config
2. Agent inspects game files, identifies engine and graphics API
3. Agent queries success database (exact match or similar games)
4. Agent pre-configures Wine settings based on engine type
5. Agent launches the game with dialog detection active
6. If the game doesn't work, agent researches fixes with structured extraction from web pages
7. Agent adapts and retries
8. Session ends with a cost summary line

The game doesn't have to actually work perfectly — what matters is that the agent follows this workflow intelligently and the new v1.1 features (engine detection, pre-config, dialog detection, structured research, budget tracking) are all visible in the output.

## Quick reference: what to grep in logs

After a session, you can check the logs:

```bash
swift run cellar log setup-cossacks-european-wars-21013
```

Look for tool calls named: `inspect_game`, `launch_game`, `trace_launch`, `list_windows`, `fetch_page`, `query_successdb`, `search_web`.
